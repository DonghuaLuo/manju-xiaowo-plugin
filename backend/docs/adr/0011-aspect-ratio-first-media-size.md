# 媒体尺寸：比例优先，分辨率只决定清晰度

manju 的分镜图和视频输出比例由项目级 `aspect_ratio` 唯一决定。`resolution`
或自定义尺寸值只表达清晰度规模，不应携带或覆盖比例。

此前各后端容易把分辨率档位直接映射成带比例的像素值，例如 `1920x1080` 会把
16:9 混入本应为 9:16 的项目。OpenAI 图片 I2I 不传 `size` 时也会让上游默认比例接管，
导致同一项目内分镜比例漂移。比例错误的图或视频对短剧工作流不可用，因此需要把比例与
清晰度解耦。

## 决策

- `aspect_ratio` 是输出比例唯一真相源。
- 分辨率档位统一规范化为短边像素：图片 `512px` / `1K` / `2K` / `4K`，视频
  `480p` / `720p` / `1080p` / `4K`。
- 自定义 `宽x高` / `宽*高` 只取短边，剥离其自带比例。
- 接受任意像素尺寸的后端通过 `backend/lib/aspect_size.py` 按比例与短边推导宽高。
- 缺分辨率但必须下传尺寸来锁定比例时，默认使用 720P 短边。

## 后端适配

- OpenAI 图片 T2I / I2I 对称下传 `size` 和 `quality`，避免 `images.edit` 丢失项目比例。
- DashScope wan、qwen-image-2.0 融合和 qwen-edit 系列使用同一短边机制，并在各自像素预算内收口。
- NewAPI 视频用 8 的倍数推导宽高，生成标准视频尺寸。
- Sora 固定合法尺寸档，不能任意传宽高；按模型能力和请求清晰度在合法档内选择最接近项目比例的尺寸。
  Sora 2 Pro 高档使用官方 `1024x1792` / `1792x1024`，不下传非官方的 1080P 像素值。
- Ark、Gemini、Grok、Vidu 等比例独立字段后端继续分别下传 `aspect_ratio` / `ratio`，不把比例压进像素尺寸。

## 影响

- 计费兜底不再从 `(resolution, aspect_ratio)` 反推 OpenAI 图片尺寸；无 usage 时落默认
  `1024x1024` 档，主路径仍以供应商返回 usage 为准。
- `gpt-image-1.5` 和 `gpt-image-1-mini` 因固定尺寸档无法满足 9:16 / 16:9 主用例，已从预设模型下架。
- 后续新增像素尺寸型后端，应优先复用 `aspect_size()` 和 `resolution_to_short_edge()`，
  不再新增静态“分辨率 × 比例”表。
